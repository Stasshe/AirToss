# 00. ビジョン

## AirToss が解く問題

近くにいる相手にファイルを渡すだけの作業が、OS が違うだけで急に難しくなる。
Apple 端末同士なら AirDrop で済むが、iPhone と Android、Mac と Windows、Linux が混ざった瞬間に、クラウドへ上げる、ケーブルを探す、メールに添付する、といった迂回が始まる。

AirToss は、この「OS が違う相手への近接転送」を一つのアプリで解く。

- クラウドサーバーを使わない
- アカウントを要求しない
- インターネット接続を要求しない
- 既設の Wi-Fi ルーターがあれば使うが、なくても動く

## AirDrop の模倣ではない

AirToss は「クロスプラットフォーム AirDrop」ではない。
AirDrop の完全な透過性（アプリを開かずに受信する体験）は、iOS のサードパーティアプリには再現できない。
この制約を隠して魔法を装うのではなく、逆に **挙動の可視性** を製品の性格にする。

利用者は常に次を知ることができる。

- 相手をどうやって見つけたか（Bluetooth）
- どの経路で転送するか（同じ Wi-Fi、直接接続、一時ネットワーク、Bluetooth 低速）
- その経路を選ぶと、いま使っているネットワークに何が起きるか（維持される、一時的に切り替わる）

## 競合と差別化

**LocalSend**：5 OS 対応、サーバー不要で完成度が高い。
ただし同一ローカルネットワークにいることが前提であり、共通の Wi-Fi がない場面（屋外、モバイル回線のみ、他人同士）では使えない。

**Flying Carpet**：共通 Wi-Fi 不要の 5 OS 対応という点で AirToss に最も近い。
ただし実機検証で次の問題を確認した。

- 起動しただけで既存の Bluetooth 周辺機器（マウス）が切断された
- GATT service を解決できず接続に失敗した（同種の報告が Issue として未解決のまま残っている）
- 何に接続しているのか、いま何が起きているのかが UI から分からない
- 転送経路の多くで、使用中の Wi-Fi 接続が切れる

AirToss の差別化は、この裏返しである。

1. **既存環境を壊さない**。Bluetooth 周辺機器と既存 Wi-Fi 接続への無影響を非機能要件として保証する。
2. **経路の自動選択**。同一 LAN、Wi-Fi Aware、Wi-Fi Direct、一時ネットワーク、BLE 低速を端末の組み合わせと能力に応じて選ぶ。可能な限りネットワークを切らない経路を優先する。
3. **可視性のある UX**。接続段階、選択経路、ネットワークへの影響を常に表示し、失敗時は次の一手を提示する。

## 成功基準

v1 が成功したと言える状態を、次のように定める。

- iPhone と Android の間で、共通の Wi-Fi なしに、アプリを開いて 30 秒以内に写真の転送が始まる
- 5 OS のどの組み合わせでも、同じ Wi-Fi にいれば同じ操作でファイルが届く
- Linux デスクトップで AirToss を起動しても、Bluetooth マウスとキーボードが一瞬も切れない
- 転送に失敗したとき、利用者が生のエラー文字列を読まされることがなく、必ず選べる次の行動が表示される

## 非目標

v1 では次を扱わない。

- 1 対多の配布、チャンク中継、イベントルーム（v2 候補）
- インターネット経由の遠隔転送
- アプリ非起動時のバックグラウンド受信
- ブラウザ版
- Wi-Fi Aware 非対応の旧型スマートフォンへの特別対応
