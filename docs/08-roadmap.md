# 08. ロードマップ

順序の意図は、**技術リスクの高い箇所を、UI やプロトコルの作り込みより先に潰す** ことにある。
Flying Carpet の失敗が示すとおり、この製品の成否は機能の数ではなく、BLE と無線経路の実機品質で決まる。
各マイルストーンには合格条件を定め、合格するまで次へ進まない。

## M0. BLE 発見と認証の PoC

対象：Linux（Ubuntu）と、iOS または Android の 1 機種。
UI は最小限のデバッグ画面でよい。ファイル転送は作らない。

合格条件：

1. 両端末でアプリを開いてから 3 秒以内に相互発見する
2. OS のペアリングダイアログを一度も出さずに GATT 接続する
3. 鍵交換と 6 桁コード表示が両端末で一致する
4. 接続と切断を 100 回連続で繰り返し、失敗が 0 回
5. その間、Ubuntu 側の Bluetooth マウスとキーボードが一度も切断されない
6. アプリ終了後、`bluetoothctl` 等で観測して scan と advertise が残っていない

条件 5 と 6 は以後のすべてのマイルストーンで回帰試験として維持する。

## M1. 同一 LAN での 5 OS 転送

BLE 認証の上に、SameLanTransport と転送プロトコル（`06`）を実装する。
Flutter の本 UI（ホーム、認証、転送、エラー）をここで作る。

合格条件：

1. 5 OS の任意の組み合わせ（10 組）で、1 GB のファイル転送が成功する
2. 転送中に送信側アプリを強制終了し、再接続後に途中から再開できる
3. 破損注入試験（チャンク改変）でハッシュ不一致が検出され、再転送で回復する
4. `02-ux.md` の全画面が実装され、経路とネットワーク影響が常時表示される

この時点で、LocalSend 相当の体験に BLE 発見（同一 LAN でも mDNS に依存しない発見）が加わった状態になる。

## M2. Wi-Fi Aware（iOS ↔ Android）

共通 Wi-Fi なし、モバイル回線のみの環境で検証する。

合格条件：

1. iPhone と Wi-Fi Aware 対応 Android の間で、共通 Wi-Fi なしにデータパスが確立する
2. 転送中、両端末のモバイル回線経由の通信（ブラウジング）が維持される
3. iOS 側の確認 UI が二重になる場合、`07-platform-notes.md` に従い UX を確定させる
4. Wi-Fi Aware 非対応 Android で、端末カードの経路見込み表示が正しく縮退する

## M3. Wi-Fi Direct（Windows ↔ Linux、Windows ↔ Windows、Linux ↔ Linux）

検証の中心は速度ではなく **並行動作** である。

合格条件：

1. 両 PC が別々の Wi-Fi に接続したまま、Wi-Fi Direct 上で TCP 接続が確立する
2. 転送中、両 PC の既存 Wi-Fi 経由の通信が維持される
3. 並行動作非対応チップで、経路が候補から外れ TempAp へ正しく落ちる
4. Windows ↔ Linux の相互運用が代表 2 機種以上で成功する。失敗が解消できない場合、この組み合わせの主経路を TempAp に確定し、組み合わせ表（`05`）を更新する

## M4. 一時 AP と経路選択 UI

macOS を含む残りの組み合わせを埋める。

合格条件：

1. 経路選択画面に成立するホスト方向と BleGatt だけが表示され、承諾なしにネットワークが切り替わることがない
2. Windows または Linux がホストする一時 AP に macOS が参加し、転送後に元のネットワークへ復帰する
3. macOS が転送中に既知 Wi-Fi へ自動復帰してしまう場合でも、再開機能で転送が完走する
4. BleGatt 低速経路で 5 MB のファイルが完走し、サイズ超過時の警告が出る

## M5. 仕上げと公開

- 権限フロー、初回起動体験、Share Sheet と「送る」メニューへの統合
- 各ストアとパッケージ（App Store、Play、winget、Homebrew、Flatpak）
- MIT ライセンスでの公開。依存ライセンスの監査（GPL 混入がないことの確認を含む)

合格条件：

1. 第三者（開発者でない利用者）が説明なしで iPhone から Ubuntu へ写真を送れる
2. `00-vision.md` の成功基準をすべて満たす

## マイルストーンをまたぐ規律

- M0 の回帰条件（周辺機器無影響、後始末）は毎マイルストーンで再実行する
- 実機検証で設計と矛盾が出たら、コードでの回避より先に該当設計書を更新し、判断の理由を記録する。設計書群が常に現状の意図を表す状態を保つ
