# 02. UX 仕様

UX はこの製品の中心である。
競合の Flying Carpet が技術的には近い構成を持ちながら使いものにならなかった直接の原因は、接続状態の不可視と、失敗時に何もできない UI にあった。
実装上の判断に迷ったら、本書の原則へ戻る。

## UX 原則

1. **いま何が起きているかを常に示す**。
   発見、認証、経路確立、転送の各段階を、利用者の言葉で表示する。無言の待ち時間を作らない。
2. **ネットワークへの影響を事前に示す**。
   経路の選択肢には必ず「既存の接続が維持されるか、切り替わるか」を添える。承諾なしに切り替えない。
3. **失敗しても行き止まりにしない**。
   すべてのエラー画面に、再試行、代替経路、詳細表示のいずれかを置く。
4. **専門用語を一次表示に出さない**。
   GATT、NAN、QUIC といった語は「詳細」の中にだけ現れてよい。一次表示は「Bluetooth で発見」「高速接続」「低速転送」の語彙で統一する。

## 基本フロー

送信側と受信側で画面を分けない。
アプリを開くと 1 つのホーム画面になり、周囲の端末が並ぶ。

```
アプリを開く
→ 周囲の端末が一覧に現れる（3 秒以内）
→ 相手を選ぶ
→ 両端末に同じ 6 桁コードが表示される
→ 双方が「一致している」を確認
→ 経路が確立し、以後このセッション内で相互にファイルを送れる
```

コード確認はセッションにつき 1 回だけ行う。
ファイルを追加で送るたびに再確認させない。

## ホーム画面

周囲の端末を、レーダーではなくリストで表示する。
各端末カードには次を出す。

```
┌────────────────────────────┐
│ 🖥 Ubuntu デスクトップ        │
│ Bluetooth で発見             │
│ 高速接続が利用できます        │
└────────────────────────────┘
```

3 行目は経路の見込みであり、次の 3 種のいずれかになる。

- 「高速接続が利用できます」（同一 LAN、Wi-Fi Aware、並行 Wi-Fi Direct のいずれかが見込める）
- 「高速接続には Wi-Fi の切り替えが必要です」
- 「Bluetooth での低速転送のみ利用できます」

自端末の表示名は初回起動時に OS のデバイス名から提案し、変更可能にする。

## 認証画面

両端末に同一の 6 桁コードを大きく表示する。

```
相手の画面に同じ番号が
表示されていることを確認してください

        4 8 2 9 1 7

   [一致している]   [キャンセル]
```

コードの意味（中間者攻撃の検出）は画面では説明しない。
「詳細」からヘルプへ誘導する。

## 経路選択画面

ネットワークを維持できる経路があるときは、この画面を出さずに自動選択する。
切り替えが必要な組み合わせ（典型は macOS と Windows/Linux が別ネットワークにいる場合）でだけ、選択肢を表示する。

```
接続方法を選んでください

● 高速転送
  この Mac の Wi-Fi が一時的に切り替わります
  転送後に自動で元に戻ります

○ 相手側で高速転送
  相手の Windows の Wi-Fi が一時的に切り替わります

○ Bluetooth 転送
  Wi-Fi はそのまま維持されます
  速度は大きく低下します（目安：10 MB で約 1 分）
```

推奨をラジオボタンの初期選択で示し、文言で煽らない。

## 転送画面

```
写真 24 枚を送信中

経路：同じ Wi-Fi（高速）
インターネット接続：維持されています

████████████░░░░░░░░  62%   48 MB / 78 MB
残り約 12 秒

              [中止]
```

進捗はファイル単位ではなくセッション全体のバイト数で示す。
経路とネットワーク影響の 2 行は転送中も常に表示する。

## 受信許可

受信側には、転送開始前に許可ダイアログを出す。

```
「Kentaro の iPhone」からファイルが届きます

  IMG_2041.jpg ほか 23 件
  合計 78 MB

   [受け取る]   [拒否]
```

同一セッション内の 2 回目以降は、設定で「このセッション中は自動許可」を選べるようにする。

## エラー設計

エラーは 3 層で扱う。

- **一次表示**：何ができなかったか、利用者の行動に結びつく形で述べる
- **選択肢**：再試行、代替経路、キャンセルのボタン
- **詳細**：折りたたみの中に、内部エラーコード、経路、失敗段階を出す

例。

```
Bluetooth で相手を発見できましたが、
高速接続を確立できませんでした

   [再試行]
   [Bluetooth 低速転送で送る]
   [キャンセル]

▸ 詳細
```

禁止事項。

- 例外メッセージやライブラリ名をそのまま一次表示に出すこと
- 「エラーが発生しました」だけで選択肢のない画面
- 失敗の責任を利用者に帰す文言（「設定を確認してください」だけの表示など）

## 文言の統一

| 内部概念 | 一次表示の語 |
|---|---|
| BLE discovery | Bluetooth で発見 |
| Same-LAN TCP | 同じ Wi-Fi（高速） |
| Wi-Fi Aware | 直接接続（高速） |
| Wi-Fi Direct（並行） | 直接接続（高速） |
| Temporary AP | 一時ネットワーク（Wi-Fi 切り替え） |
| BLE GATT 転送 | Bluetooth（低速） |

Wi-Fi Aware と並行 Wi-Fi Direct を利用者向けに区別しない。
どちらも「既存の接続を維持したまま直接つながる高速経路」であり、区別は利用者の判断に影響しないからである。
