# 01. 要件

## 対象環境

利用者層は現行世代の端末を持つと想定する。
旧機種のための妥協を設計に持ち込まない。

| プラットフォーム | 最低条件 | 備考 |
|---|---|---|
| iOS | iPhone 13 以降、iOS 26 以降 | Wi-Fi Aware framework の利用が前提 |
| Android | Android 12（API 31）以降 | Wi-Fi Aware は端末依存。実行時に判定する |
| Windows | Windows 11 | Bluetooth と Wi-Fi アダプター搭載を前提とする |
| macOS | 現行 −2 メジャーバージョン | 同上 |
| Linux | BlueZ 5.x、NetworkManager 環境のデスクトップ | 同上。Ubuntu LTS を基準ディストリビューションとする |

PC は無線 LAN カードと Bluetooth を持つ前提でよい。
ただし Wi-Fi Direct の並行動作と AP モードはチップとドライバーに依存するため、能力は実行時に検出し、使えない場合は経路候補から外す（`05-transport.md`）。

## 機能要件

- **FR-1 発見**：アプリを開いた端末同士が、同一ネットワークにいなくても、BLE によって互いを発見できる。
- **FR-2 認証**：OS の Bluetooth ペアリングを使わず、両端末に表示される 6 桁の確認コードで相手を認証できる。
- **FR-3 経路選択**：端末の組み合わせと実行時能力に基づき、転送経路を自動選択する。ネットワークの切り替えが必要な経路は、利用者の明示的な承諾なしに使わない。
- **FR-4 転送**：ファイル（複数可）とテキストを送信できる。受信側は受け取る前に、送信者、ファイル名、サイズを確認して許可または拒否できる。
- **FR-5 再開**：転送が中断した場合、同一セッション内で途中から再開できる。
- **FR-6 検証**：受信ファイルはハッシュ検証され、破損があれば利用者に通知される。
- **FR-7 双方向**：どちらの端末も送信側にも受信側にもなれる。役割の固定はない。

## 非機能要件

- **NFR-1 周辺機器無影響**：AirToss の起動、発見、転送のいずれの段階でも、接続済みの Bluetooth 周辺機器（マウス、キーボード、イヤホン）を切断させない。
  このために、Bluetooth アダプターの再起動、電源操作、既存ペアリングの削除、`bluetoothd` の操作を一切行わない。
- **NFR-2 ネットワーク無影響の原則**：既存の Wi-Fi 接続を維持できる経路が存在する限り、それを選ぶ。切り替えが避けられない組み合わせでは、切り替わる側と影響を事前に明示し、承諾を得る。
- **NFR-3 発見速度**：両端末でアプリを開いてから、相手が一覧に表示されるまで 3 秒以内。
- **NFR-4 転送性能**：Wi-Fi 系経路で実効 10 MB/s 以上を目標とする（環境依存のため保証値ではなく設計目標）。
- **NFR-5 後始末**：交渉完了後および転送完了後、BLE の Scan と Advertising を即座に停止する。アプリ終了後に無線動作を残さない。
- **NFR-6 プライバシー**：BLE Advertising に恒久的な識別子を載せない。端末 ID はセッションごとに生成する。利用ログを外部送信しない（そもそも外部サーバーが存在しない）。
- **NFR-7 セキュリティ**：転送経路によらず、アプリケーション層でエンドツーエンド暗号化する。鍵は確認コードで認証された BLE ハンドシェイクから導出する（`04-discovery-auth.md`）。
- **NFR-8 エラー提示**：生の例外やスタックトレースを一次表示しない。利用者向けの説明と選択肢を表示し、内部情報は「詳細」に隔離する（`02-ux.md`）。

## 明示的にしないこと

次の否定は要件である。

- OS の Bluetooth ペアリングダイアログを出さない
- 常駐サービスやバックグラウンドスキャンを行わない
- 転送内容やメタデータをディスク上に平文キャッシュとして残さない（受信ファイル本体を除く）
